# Pi Study Helper 文档

本目录按“设计—形态—计划—审查—归档—参考”管理当前事实、未来工作与历史材料。

## 第一次阅读

1. [项目主要目标](设计/该项目主要目标.md)
2. [架构边界](设计/架构边界.md)
3. [当前形态](形态/README.md)
4. [实施计划](计划/README.md)
5. [审查与验收](审查/README.md)

## 目录职责

| 目录 | 只保存什么 |
| --- | --- |
| [设计](设计/README.md) | 稳定目标、边界、已接受的架构决策 |
| [形态](形态/README.md) | 当前已经实现到什么程度 |
| [计划](计划/README.md) | 尚未完成的路线、任务、工时和调用预算 |
| [审查](审查/README.md) | 验收证据、风险、SDK 反馈与发布门禁 |
| [归档](归档/README.md) | 已失效或已完成的历史材料 |
| [参考](参考/README.md) | 外部仓库事实快照，不代表本项目契约 |

跨设计主题且难以逆转的决策统一保存在 [ADR](adr/README.md)；当前设计入口见[设计文档](设计/README.md)。

## 维护规则

- 设计文档描述“为什么和边界是什么”，不伪装成当前实现。
- 形态文档只描述已经由代码或测试证明的事实。
- 计划完成后移入归档，并在形态文档写入新事实。
- 参考文档注明来源路径和核对日期；来源变化时重新核对。
- SDK 问题必须在审查文档中附复现条件和证据。

`draft.md` 是本轮文档建设的原始要求，完成本体系后仅作为历史输入保留。
