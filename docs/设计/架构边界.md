# 架构边界

## 设计原则

确定性责任由代码承担，语义性责任由 Agent 承担，跨阶段变化由 Loop Graph 明确表达。

| 责任                                | 实现方式                            | 原因                            |
| ----------------------------------- | ----------------------------------- | ------------------------------- |
| 输入选择、TUI 展示、答案采集        | 代码侧业务能力 + TUI gateway        | 交互必须一致且可测试            |
| Profile 读取、路径安全、schema 校验 | 代码 service                        | 不能信任模型自报                |
| 题目生成                            | Agent Run + output schema           | 需要语义生成能力                |
| 判题与解析                          | Agent Run + output schema           | 需要理解自由文本答案            |
| 归档、进度和会话状态                | 代码 service                        | 必须幂等、可验证                |
| 深度讨论                            | 可选 Agent Run                      | 只在用户需要时发生              |
| 学习情况总结                        | 必需 Agent Run + 代码持久化         | 总结需要语义归纳，保存必须可靠  |
| 路由                                | Edge guard + priority/custom router | 业务迁移不能隐藏在 Agent 文本中 |
| 观察与验收                          | Mechanism + trace +代码校验         | 自报结果与可信证据分离          |

## 学习会话图

```text
prepare_session (code)
  → present_material (code)
  → generate_question (agent)
  → collect_answer (code)
  → grade_answer (agent)--(错误)->collect_answer(code)
  ()→ persist_attempt (code)
  → choose_action (code)
       ├─ next/show material ─────────────→ generate_question
       ├─ discuss → discuss_answer (agent) → choose_action
       └─ summary/exit → summarize_session (agent)
                           → persist_summary (code)
                           → END
```

`persist_summary` 成功是正常 `END` 的必要条件。取消或基础设施错误走 `cancelled`/`failed` 路径；进程级异常由逐题归档和下次启动恢复检查处理。

## 推荐图边界

- `study_session_graph`：顶层学习会话，拥有循环和正常结束约束。
- `study_turn_graph`：可选 `compose` 子图，表达一题从准备到归档的阶段；只有在上下文归约验证通过后采用。
- `discussion_graph`：用 `call` 接收明确题目、答案和判题结果，避免污染主图工作记忆。
- `session_summary_graph`：用 `call` 接收代码聚合后的会话事实，不读取完整对话历史。
- `profile_build_graph`、`profile_revise_graph`：第二里程碑的独立顶层图。

首个 walking skeleton 可以先使用一张顶层图，待 trace 证明边界稳定后再抽子图，避免一开始同时调试业务与 compose 归约。

## SDK 接入边界

当前 SDK 的 `NodeContext.callTool()` 只有接口签名，`PiNodeContext` 的实现会直接抛错。根因是 pi `ExtensionAPI` 只能注册、列举和启停工具，不能通过公开 API 执行已注册工具；`getAllTools()` 也不会返回 `execute`。因此 `callTool()` 不是本项目可以依赖或等待补齐的执行路径。

本项目采用代码侧业务能力：

1. 业务 extension 自行注册交互命令，例如 `/study`。
2. 命令 handler 创建本次会话的 `StudySessionServices`，包含 TUI gateway、Profile reader、archive repository 和 clock/id generator；这些接口就是代码侧业务能力。
3. graph factory 通过闭包接收这些 service，代码节点直接调用其底层实现，不经过 pi 工具注册表。
4. handler 使用 `LoopGraphExtension.executeGraph()` 程序化运行无公开 invocation 的图。
5. 若某项能力也需要开放给 Agent，则额外注册一个薄 pi tool adapter；adapter 和代码节点复用同一底层函数，不能互相调用。

此方案必须在第一阶段通过真实 pi TUI spike 验证 ExtensionContext 生命周期以及 Agent Run 与长命令 handler 的协作。即使未来 pi 增加 `invokeTool()`，是否改用它也需要重新评估；首版不以此为前置条件。

### 代码调用与工具调用的语义差异

代码侧直接调用不会自动获得 pi tool 的参数 schema 校验、`tool_result` 事件、工具渲染和 Mechanism 工具 Hook。业务能力必须自行提供输入校验、取消信号、错误归一化和观测事件；测试也应直接验证其副作用和返回值。不能把“复用了工具底层函数”表述为“调用了 pi tool”。

## 状态通道

- `background`：稳定的会话标识、Profile 标识和用户策略。
- `NodeInput`：本次访问所需的一次性输入。
- `ContextFrame`：只保留后续 Agent 真正需要的业务摘要，禁止塞入完整题目历史。
- 产品 repository：答题、会话和学习画像的可信长期数据源。
- `Mechanism state`：只保存横切观察或约束状态，不代替业务 repository。

总结节点从 repository 读取结构化会话事实。它不依赖模型上下文中恰好还保留全部题目，因此能抵抗 compaction。

## SDK 已知范围

- 单 Agent 串行编排；一张图同时只推进一条路径。
- 同一个 LoopGraphExtension instance 不支持并发 root `executeGraph()`。
- 不支持多 Agent 通讯与 fork/join。
- 不支持未完成图的跨进程恢复。
- 一个节点只有一个 `skill` 引用；全局规则与阶段规则需要合成 skill 内容或通过 renderer/provider 解决。
- `callTool()` 当前是不可用的占位接口；纯代码节点直接调用代码侧业务能力。

这些限制是首版设计输入，也是 SDK 验证清单的一部分。
