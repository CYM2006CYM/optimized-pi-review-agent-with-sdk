## 工作方式

一个“开发轮次”表示我完成一次闭环：

```text
读取当前代码事实
→ 实现一个有边界的功能或修复
→ 添加或更新测试
→ 执行相关验证
→ 更新形态、计划或审查中的必要事实
```

一个轮次不等于一次工具调用，也不等于一次底层模型请求。复杂功能可能连续占用多个对话轮次；估算的作用是控制实施顺序，不是承诺日历时间。

用户主要负责：

- 需要真实 TUI 操作时执行人工测试并返回现象。
- 对无法从现有代码和参考资料推导的产品定义作最终决定。
- 决定是否接受阶段性产品行为。

其余设计、实现、测试、诊断和必要文档同步由我负责。

## 里程碑一目标

里程碑一是一个完整纵向产品，不再把基础 Profile 创建和修订推迟到另一个里程碑。完成范围包括：

- Profile family 生命周期。
- Markdown/txt 自动创建 canonical Profile。
- 基于 active 创建 revision draft、反复修改、质量审查、确认或放弃。
- 三种任务驱动学习方式。
- 出题、重答、放弃、解析、讨论和题后功能菜单。
- 逐题记录、异常中断和强制会话总结。
- 用户手动生成学习画像，成功后消费 pending 记录。
- 真实 pi、长会话和关键失败分支验收。

## 剩余实施轮次

### P0：交互式学习闭环验收与修复

预计：1～3 个开发轮次。

1. 接收用户执行人工清单后的结果和测试数据路径。
2. 检查 session、attempt、summary 和 lifecycle trace。
3. 修复阻塞、菜单、状态或持久化问题。
4. 对关键修复增加自动测试，并让用户只复测失败分支。

退出条件：

- 三种学习方式均能从选择进入总结。
- 正确、错误重答、讨论、放弃、换范围、提高难度和中断行为都有证据。
- 正常结束存在总结；异常结束为 interrupted。

### P1：学习状态机加固

预计：2～4 个开发轮次。

1. 从 extension handler 中提取可测试的学习会话控制器和 TUI gateway 接口。
2. 为正确、错误、重答、放弃、讨论和题后菜单添加状态机测试。
3. 增加启动时遗留 running 会话处理和补总结入口。
4. 明确取消、空输入、模型失败、总结失败和持久化失败的用户提示。

退出条件：主要交互分支不再只能依赖人工点击验证；每题保存和最终总结门禁有自动回归测试。

### P2：手动学习画像

预计：2～3 个开发轮次。

1. 在 Profile 管理入口列出未消费的 pending 学习批次。
2. 建立学习画像 Agent 图和严格输出校验。
3. 用户确认后写入 `learning_profile.json`。
4. 仅在画像成功写入后归档被消费批次。
5. 验证失败、取消和部分移动回滚。

退出条件：用户可以手动生成画像；失败时 pending 记录保持不变。

### P3：原始资料自动创建 Profile

预计：3～6 个开发轮次。

1. 代码盘点 Markdown/txt、计算 hash、限制路径并按上下文预算分批。
2. 建立构建 job/checkpoint，不依赖恢复 SDK 图会话。
3. Agent 提取资料结构并生成章节、卡片、考点和来源映射。
4. 代码写入唯一 draft，并持续执行 canonical schema 校验。
5. 生成结构质量报告和语义质量报告。
6. 用户确认后启用；不满意时保留修改循环或明确放弃。

退出条件：给定一组中型 Markdown/txt，可以生成并启用一个新的 canonical Profile。

### P4：Profile 修订闭环

预计：2～4 个开发轮次。

1. 从 active 复制 revision draft。
2. 收集用户修订意见并计算影响范围。
3. 只重建受影响单元，保留无关内容。
4. 重新执行结构和语义质量审查。
5. 用户确认时归档旧 active；不满意时继续修改或放弃 draft。

退出条件：修订永远不直接写 active，且确认、继续修改、放弃三个出口都有测试证据。

### P5：里程碑一稳定验收

预计：3～5 个开发轮次。

1. 10 题以上长会话与 repository 总结一致性。
2. compaction、取消、Agent 超时、schema 重试和 TUI 关闭。
3. Profile 构建/修订 job 的取消和 checkpoint 重启。
4. 第二模型提供方验证，或明确记录只支持已验证提供方。
5. 汇总 Agent Run、节点访问、失败原因和可观测性缺口。

退出条件：[审查门禁](../审查/README.md)中所有里程碑一必需项通过。

## 总体剩余轮次

从当前状态估计，里程碑一还需要约 10～20 个有效开发轮次。

这个范围允许：

- 人工测试首次发现若干交互问题。
- Profile 构建需要 1～2 次 schema/提示词修正。
- 稳定阶段出现至少一个 SDK 或模型相关诊断轮次。

如果人工 `/study` 一次通过且 Profile 构建输出稳定，轮次会接近下界；如果需要重构 TUI 状态机或 SDK 出现可复现缺口，则接近上界。

## 运行时 Agent Run 预算

学习会话的实际 Agent Run 为：

```text
Agent Runs = 出题数 Q + 实际答案提交数 A + 讨论轮数 D + 总结次数 S
```

- 首次回答即完成：每题 1 次出题 + 1 次判题。
- 每次重新提交答案增加 1 次判题。
- 主动放弃不调用判题 Agent，直接使用题目已有答案和解析。
- 每轮开放讨论增加 1 次 Agent Run。
- 正常结束至少增加 1 次总结。
- 菜单中查看当前总结每次再增加 1 次总结 Run。

典型示例：

| 场景 | Agent Runs |
| --- | ---: |
| 1 题首次答对后结束 | 3 |
| 1 题答错一次、重答正确后结束 | 4 |
| 上述过程再讨论一轮 | 5 |
| 10 题均首次完成、无讨论 | 21 |

这只统计 SDK Agent Run，不等于底层 HTTP 请求数。completion validator 驳回、工具 turn 或模型重试可能让一次 Run 产生多次请求。

## 需要用户决定的情况

只有以下情况应暂停实现并询问用户：

- 两种合理行为会显著改变产品体验或数据格式。
- 操作会删除、覆盖或发布用户真实资料。
- 需要扩大到当前里程碑之外的产品范围。
- 同一阻塞经过安全诊断仍无法从代码、测试或参考实现确定答案。

其余实现细节由我作出可逆决定并继续推进。
